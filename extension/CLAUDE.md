## Commands

```bash
npm run dev        # Vite watch build → dist/ (load this folder as unpacked extension)
npm run build      # Production build
npm run type-check # tsc --noEmit (no tests exist yet)
```

Reload the extension in Chrome after each build: `chrome://extensions` → click the refresh icon on the card.

## Environment

Create `extension/.env` with:
```
VITE_API_BASE=http://localhost:3000
```

`VITE_API_BASE` is read in `background/index.ts` via `import.meta.env.VITE_API_BASE`. If port 3000 is occupied by another app, set a different port here and in `server/.env` (`PORT=xxxx`).

## Source layout

```
src/
  manifest.ts          # Typed MV3 manifest (via @crxjs/vite-plugin defineManifest)
  background/index.ts  # Service worker — the single orchestrator
  content/             # One script per platform (youtube/tiktok/instagram/facebook)
  sidepanel/
    main.tsx           # React entry
    App.tsx            # Top-level view router (main / memory / auth)
    store/index.ts     # Zustand store — all UI state
    hooks/
      useAuth.ts           # Supabase session → store.user + chrome.storage.local
      usePlatformListener.ts # background messages → store.platformState
      useLibrary.ts        # Supabase packs/collections → store
    services/api.ts    # Thin wrappers: sendMessage(START_EXTRACTION) etc.
    components/        # Pure React components, no side-effects
public/                # icon16/48/128.png (generated by scripts/generate-icons.mjs)
```

`@shared/*` resolves to `../shared/*` — types are defined there, never duplicated here.

## Key architectural constraints

- **No `fetch` in the side panel.** All server calls go through `background/index.ts` via `chrome.runtime.sendMessage`. `services/api.ts` is the only place that sends messages; components call into it.
- **Side-effects only in hooks and background.** React components are stateless — they read from the Zustand store and call store actions or `services/api.ts`.
- **Live capture is opt-in.** `START_LIVE_CAPTURE` is sent to the content script only after the user clicks Extract and the strategy is `live`. Never start it automatically.
- **Auth token lives in `chrome.storage.local`** under key `supabase_token`. The background worker reads it before every `/extract` request; the side panel writes it via `useAuth`.

## MV3 gotchas

- `chrome.runtime.onMessage` can only have one `return true` per listener for async responses. Two separate `addListener` calls are used in `background/index.ts` — one for content-script messages (no async response needed), one for side-panel messages (returns `true` for `GET_CURRENT_PLATFORM`).
- The service worker can be terminated at any time. `tabStates` (in-memory Map) will be lost on worker restart — this is a known limitation, not a bug.
- `crx()` handles content-script and service-worker bundling automatically via the manifest; no manual entry points in `vite.config.ts` are needed.
